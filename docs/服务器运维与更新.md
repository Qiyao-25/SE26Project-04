## 2. 服务器注意事项

### 2.1 网络与访问

- `10.x` 浮动 IP **不是全球公网**；同学需在校园网或 VPN 下访问。
- 安全组至少放行 **22 / 80**（HTTPS 再放 443）。
- 后端只监听本机 `8000`，由 Nginx 反代 `/api` 与 `/health`，不要把 8000 直接暴露到公网。
- 订阅同步、LLM 需主机 **出站** 可访问 arXiv 与 LLM API。

### 2.2 Python 版本

- 项目 `requires-python = ">=3.11"`。Ubuntu 22.04 默认常为 3.10，必须用 **python3.11** 建 venv。
- 一键脚本 `deploy/deploy-on-host.sh` 会安装 deadsnakes 的 3.11，并固定使用 `/usr/bin/python3.11`。
- **不要**在已 `activate` 旧 `.venv` 的 shell 里跑安装脚本（否则可能解析到即将被删除的解释器）。先 `deactivate` 再执行。

### 2.3 密钥与环境变量

必须在服务器编辑 `/opt/papermate/backend/.env`：

| 变量 | 说明 |
|------|------|
| `PAPERMATE_ENV=prod` | 生产模式 |
| `PAPERMATE_AUTH_SECRET` | 长随机串；修改后已登录会话失效 |
| `PAPERMATE_LLM_API_KEY` | 摘要 / 问答等依赖 |
| `PAPERMATE_CORS_ORIGINS` | 含实际访问来源，如 `http://10.119.9.119` |
| `PAPERMATE_ENABLE_DOCS` | 验收可临时 `true`，正式建议 `false` |

```bash
# 生成 AUTH_SECRET 示例
SECRET=$(openssl rand -hex 32)
sed -i "s|^PAPERMATE_AUTH_SECRET=.*|PAPERMATE_AUTH_SECRET=${SECRET}|" /opt/papermate/backend/.env
systemctl restart papermate-backend
```

`.env` 权限建议 `chmod 600`，且永不进仓库。

### 2.4 数据与备份

| 类型 | 路径 / 命令 |
|------|-------------|
| SQLite | 停服务后复制 `data/prod.db` |
| `.env` | 单独备份到安全位置（含 Key） |
| PostgreSQL | `pg_dump`（正式库推荐） |

更新前建议至少备份 `.env` 与数据库文件。

### 2.5 常用运维命令

```bash
systemctl status papermate-backend --no-pager
journalctl -u papermate-backend -n 100 --no-pager
systemctl restart papermate-backend
nginx -t && systemctl reload nginx
curl -fsS http://127.0.0.1:8000/health
curl -fsS -o /dev/null -w "%{http_code}\n" http://127.0.0.1/health
```

可选重新导入种子（会按脚本逻辑创建/更新论文，慎用生产库）：

```bash
cd /opt/papermate/backend
. .venv/bin/activate
python -m scripts.import_seed --seed /tmp/papermate/seed.json
```

## 3. 源码变更后的更新流程

原则：**本机改代码 → 打包 → 上传 → 云主机安装脚本更新 → 恢复/核对 `.env` → 重启冒烟**。不要长期只在服务器上改源码，以免与 Git 漂移。

### 3.1 本机打包

在仓库 `SE26Project-04` 根目录（或上层 `project` 工作区）：

```powershell
# 前端有改动时先构建
cd UIPrototype\frontend
npm ci
npm run build
cd ..\..\..

# 打包到仓库外工作目录（推荐，不污染源码树）
python deploy\pack.py --out ..\ppp\deploy-artifacts\packages --skip-frontend-build
```

也可输出到 `dist/packages/`（`python deploy/pack.py` 默认行为）。

产物：

- `papermate-frontend-*.tar.gz`
- `papermate-backend-*.tar.gz`
- `papermate-deploy-config-*.tar.gz`（可选）

### 3.2 上传到服务器

```powershell
$env:SSH_HOST = "10.119.9.119"
$env:SSH_USER = "root"
# 包目录：默认尝试 ppp/deploy-artifacts/packages，可用 PACKAGES_DIR 覆盖
powershell -File deploy\upload-packages.ps1
```

脚本会上传 tar.gz、`deploy-on-host.sh`，若存在 `seed.json` 一并上传到 `/tmp/papermate/`。

### 3.3 云主机执行更新

```bash
# 强烈建议：更新前备份
cp /opt/papermate/backend/.env /root/papermate.env.bak
cp -a /opt/papermate/backend/data /root/papermate-data.bak 2>/dev/null || true

deactivate 2>/dev/null || true
sed -i 's/\r$//' /tmp/papermate/deploy-on-host.sh
bash /tmp/papermate/deploy-on-host.sh
```

`deploy/deploy-on-host.sh` 在替换后端目录时会尝试：

1. 保留已有 `.env`（若新包无 `.env` 则从旧目录拷回）
2. 保留 `data/`（SQLite 等）

仍建议保留 `/root/papermate.env.bak`；若发现配置丢失：

```bash
cp /root/papermate.env.bak /opt/papermate/backend/.env
chown papermate:papermate /opt/papermate/backend/.env
systemctl restart papermate-backend
```

### 3.4 更新后冒烟

1. `curl -f http://127.0.0.1:8000/health`
2. 浏览器打开 http://10.119.9.119/ ：登录、工作台、论文详情、问答
3. 若只改了后端依赖或迁移：确认 `alembic` 已在脚本中执行成功
4. 若改了前端环境变量：必须 **重新 `npm run build` 再打包**，仅换后端包无效

## 4. 仓库内外文件分工

| 位置 | 用途 |
|------|------|
| `SE26Project-04/deploy/` | **入库**：Nginx/systemd 样例、打包脚本、一键安装与上传脚本 |
| `SE26Project-04/docs/` | **入库**：部署说明、运维更新、核对清单 |
| `ppp/deploy-artifacts/` | **工作副本**：打好的 tar.gz、本地冒烟日志、含密钥的临时 env；**默认不进本仓库** |

`ppp` 中的安装包、日志、真实 Key **不要**同步进 Git。脚本与文档以本仓库 `deploy/`、`docs/` 为准；本地可用 `ppp` 仅作产物暂存。

## 5. 故障速查

| 现象 | 处理 |
|------|------|
| `requires a different Python: 3.10 ... not in '>=3.11'` | 安装 python3.11，用 `/usr/bin/python3.11 -m venv` 重建 |
| `.../.venv/bin/python3.11: No such file` | `deactivate` 后重跑；脚本须用系统 `/usr/bin/python3.11` |
| 页面开、问答失败 | 检查 `PAPERMATE_LLM_API_KEY` 并重启服务 |
| 校外打不开 | 连校园 VPN，或确认是否仍在用 `10.x` 浮动 IP |
| 登录全失效 | 刚改过 `AUTH_SECRET`，属预期，重新注册/登录 |
| Windows 上传的 `.sh` 异常 | `sed -i 's/\r$//' deploy-on-host.sh` |

## 6. 调试：手动触发定时抓取（可删）

设置页「抓取资源配置」下有橙色 **调试 · 定时抓取** 卡片（需构建时 `VITE_ENABLE_CRAWL_DEBUG=true`，后端 `PAPERMATE_ENABLE_CRAWL_DEBUG=true`）。**仅管理员**可调用后端接口。

删除方式：

1. 删除 `UIPrototype/frontend/src/debug/`，并去掉 `SettingsPage.jsx` 中对应 import/组件
2. 删除 `backend/app/api/debug_crawl.py`，并去掉 `main.py` 中 DEBUG 注册
3. 关闭上述两个环境变量后重新打包部署

## 7. 演示架构边界（进程内解析）

当前正式路径：

- 点击「解析」→ `POST /api/papers/{id}/parse`（需登录）→ FastAPI `BackgroundTasks` 进程内执行 `parse_agent_runner`
- 定时抓取在同一 Uvicorn 进程的 lifespan 循环中运行

已知限制（演示可接受，生产需升级）：

1. **进程崩溃会丢失执行中的解析任务**（queued/running 需依赖后续 `recover-stale` 或人工重试）
2. **Uvicorn 必须 `--workers 1`**，否则会启动多个 crawl scheduler
3. **未使用 Redis/Celery/Outbox**；PaperPipeline Worker 为可选补偿路径，主路径不依赖
4. PDF **仅临时下载解析后删除**，不落盘；`paper_contents.storage_path` 为预留字段
5. `embedding` 字段为预留；检索仍为文本块词法匹配，无向量/BM25/reranker

生产启动校验：`PAPERMATE_ENV=prod` 时必须设置强 `AUTH_SECRET`、`WORKER_TOKEN`，且 `CORS_ORIGINS` 非空；默认关闭 `/docs`。
