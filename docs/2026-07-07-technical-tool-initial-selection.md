# 7月7日技术工具初选记录

## 一、任务背景

项目名称：A7 PaperMate：ArXiv论文阅读工具

本迭代目标是完成“界面原型迭代”，主要交付静态主要界面、需求文档、use-case模型和评审记录。因此，7月6日技术负责人的重点不是搭建完整后端系统，而是先判断使用什么工具能够最快、最稳定地支撑界面原型展示和小组评审。

## 二、技术负责人当日目标

1. 梳理可用于界面原型制作的工具和技术方案。
2. 判断各方案是否适合一周内完成静态原型交付。
3. 初步确定推荐方案和备用方案。
4. 记录后续迭代需要继续验证的技术风险。

## 三、候选方案

### 方案一：Figma 原型

优点：

- 上手快，适合制作低保真和高保真界面。
- 支持页面跳转和简单交互，便于评审展示。
- 不需要配置开发环境，适合非前端成员参与。

缺点：

- 不能直接体现真实代码结构。
- 后续如果进入代码开发阶段，需要重新实现页面。
- 对动态数据、加载状态、错误状态的表现能力有限。

适用场景：

- 时间紧，主要目标是展示界面结构和页面流程。
- 组员更熟悉设计工具，不熟悉前端开发。

### 方案二：HTML + CSS + 少量 JavaScript 静态原型

优点：

- 不需要复杂构建工具，文件可直接运行或用简单本地服务预览。
- 能够更真实地表现网页布局、响应式效果和基础交互。
- 后续可以作为前端开发的页面参考。
- 适合本迭代“静态主要界面”的交付要求。

缺点：

- 需要统一样式规范，否则页面容易风格不一致。
- 复杂交互需要额外编写 JavaScript。
- 如果页面数量较多，纯静态文件维护成本会上升。

适用场景：

- 需要快速做出可展示的网页原型。
- 页面以展示为主，暂时不接真实后端接口。

### 方案三：React + TypeScript + Vite

优点：

- 工程结构清晰，适合后续继续开发。
- 组件化能力强，适合复用搜索框、卡片、标签、问答面板等界面元素。
- TypeScript 有利于减少后续开发中的数据结构错误。

缺点：

- 需要配置环境和学习基础框架，第一周时间成本较高。
- 对当前“界面原型迭代”来说可能偏重。
- 如果组员不熟悉 React，容易把时间消耗在框架问题上。

适用场景：

- 团队已有 React 基础。
- 希望第一轮原型直接演进为后续前端代码。

### 方案四：React + Ant Design 或其他组件库

优点：

- 表格、标签、抽屉、弹窗、菜单、输入框等组件现成可用。
- 页面风格统一，适合后台管理类或工具类系统。
- 能快速搭出论文列表、筛选、详情、问答等界面。

缺点：

- 依赖安装和工程配置较多。
- 默认视觉风格可能偏管理系统，需要适当调整才能符合论文阅读工具定位。
- 学习成本高于纯 HTML/CSS。

适用场景：

- 后续明确要做 Web 应用，并且团队能承担框架学习成本。
- 需要较多表格、表单、标签、状态组件。

## 五、初步推荐结论

本迭代推荐采用：

```text
HTML + CSS + 少量 JavaScript 静态原型
```

推荐理由：

1. 本迭代目标是“界面原型”，不要求真实后端、数据库或模型调用。
2. HTML/CSS 能够快速表现论文库、论文详情、结构化摘要、智能问答、个人画像等核心页面。
3. 相比 Figma，HTML 原型更接近真实网页，便于后续前端开发参考。
4. 相比 React，HTML/CSS 的环境成本更低，更适合一周内稳定交付。
5. 少量 JavaScript 可用于页面切换、筛选模拟、问答面板展开等简单交互。

备用方案：

```text
Figma
```

如果代码原型推进不顺，或组员前端基础不足，可以切换到 Figma，确保先完成可评审的界面表达。

后续方案：

```text
React + TypeScript + Vite + Ant Design
```

该方案更适合迭代2或后续正式开发阶段使用，可在技术原型阶段验证组件化、接口联调和状态管理。

## 六、页面原型技术范围

7月7日暂定本迭代原型至少覆盖以下页面：

1. 首页 / 仪表盘：展示最近论文、推荐论文、阅读进度、系统状态。
2. 论文列表页：支持关键词搜索、标签筛选、排序和论文卡片展示。
3. 论文详情 / 阅读页：展示标题、作者、摘要、PDF/HTML 阅读区、章节导航。
4. 结构化摘要页：展示研究问题、方法、贡献、实验结果、局限性。
5. 智能问答页：展示用户问题、AI回答、引用出处和置信提示。
6. 个人画像页：展示研究兴趣、阅读历史、收藏主题和推荐方向。
7. 订阅 / 推荐页：展示主题订阅、推荐论文和更新状态。

## 十三、后续技术风险
风险	说明	当前处理方式
arXiv 数据获取风险	后续真实开发需要调用 arXiv API，可能受到网络和接口字段影响	本迭代使用模拟数据
PDF/HTML 解析风险	论文格式复杂，标题、公式、图表和引用不一定能稳定解析	本迭代表现解析后的静态结果
LLM 问答风险	AI 回答存在准确性、引用定位和响应时间问题	本迭代使用静态问答案例
知识库检索风险	后续需要验证向量检索、关键词检索和引用追踪效果	本迭代只展示检索入口和结果样式
页面范围过大风险	页面过多可能导致原型无法按时完成	优先完成核心页面，复杂功能只保留入口
样式不一致风险	多人协作时可能出现页面风格不统一	制定统一目录结构、命名规范和基础样式
