# 7月8日用例可实现性与技术风险检查

## 一、今日任务

今日任务是根据 use-case 模型和页面流程草图，检查各用例在当前迭代中是否可以通过界面原型表达，并识别后续真实开发阶段可能存在的技术风险。

本次检查重点不要求真实实现 arXiv 抓取、PDF 解析、LLM 问答或多 Agent 后台服务，而是判断：

1. 用例是否能在本迭代的静态原型中表现出来。
2. 用例是否需要后续真实接口、算法或后台服务支持。
3. 是否存在技术风险、范围过大或用例边界不清的问题。

## 二、检查依据

本次检查依据包括：

- use-case 模型图。
- 页面流程草图。
- 7月7日技术选型结论：本迭代优先采用 HTML + CSS + 少量 JavaScript 静态原型。
- 本迭代目标：完成可评审的静态主要界面和主要流程，不接入真实后端服务。

|---|---|---|---|---|---|
| R1 | arXiv API 调用风险 | 高 | 网络、接口字段、调用限制会影响论文检索和抓取 | 使用 Mock 论文数据 | 迭代2编写接口调用 Demo |

## 七、用例模型中发现的问题

1. 部分用例过于偏技术实现，例如“读取 Agent 任务队列”“调用外部检索接口”“解析论文 PDF 内容”。这些更像系统内部能力，不完全是用户直接操作，应在文档中标明为后台支撑用例。
2. 普通用户、研究型用户、主题追踪用户和管理员之间的权限边界还不够清楚。建议后续补充不同角色能访问哪些页面。
3. “论文检索”和“手动触发抓取论文”的关系需要进一步明确。论文检索是用户主动查询，抓取论文更像后台任务或订阅任务，不建议完全混在一个流程中。
4. “查看论文结构化详情”依赖“解析论文 PDF 内容”，这部分在本迭代只能用模拟结果展示，不能承诺真实解析效果。
5. “进行带出处智能问答”是核心亮点，但技术风险较高，应在原型中明确显示“回答出处”“引用片段”“置信提示”，避免看起来像普通聊天功能。
6. “多 Agent 流水线状态”适合做成演示者或管理员页面，不建议作为普通用户主流程。
7. 页面流程草图中的页面较多，若时间不足，应优先保证首页、论文检索页、论文详情页、智能助手页和订阅设置页。

## 八、优先级建议

### 第一优先级

必须在本迭代原型中完成：

- 登录 / 注册页。
- 工作台 / 首页。
- 论文检索与预测页。
- 论文详情页。
- 智能助手页。
- 抓取源配置与订阅设置页。

### 第二优先级

尽量完成，用于体现项目特色：

- 知识图谱与脉络分析页。
- 学习空间 / Wiki 页。
- 收藏夹与笔记评论。
- 阅读历史与概念速查。

### 第三优先级

如果时间不足，可以只保留入口或静态状态：

- 多 Agent 监控台。
- 任务队列与处理进度。
- 异常日志与人工修正。
- 测试页 / 调试页。

## 九、当前迭代实现建议

本迭代采用 HTML + CSS + 少量 JavaScript 静态原型时，建议这样处理各类功能：

| 功能类型 | 当前迭代实现方式 |
|---|---|
| 用户登录 | 静态登录表单，点击后跳转首页 |
| 论文检索 | 使用固定论文数据展示搜索结果 |
| 手动抓取 | 按钮点击后显示“抓取中 / 已完成”的状态变化 |
| 论文详情 | 静态展示论文标题、作者、摘要、标签和结构化分析 |
| PDF 解析 | 不做真实解析，只展示解析后的样例结果 |
| 智能问答 | 写死 2 到 3 个问答样例，展示出处和引用片段 |
| 知识图谱 | 使用静态节点图或简化关系图 |
| 订阅设置 | 静态表单展示关键词、频率和推荐范围 |
| Agent 监控 | 用状态卡片展示任务队列、处理进度和异常日志 |

## 十、检查结论

经过检查，当前 use-case 模型中的主要用户用例基本可以通过静态界面原型表达，适合继续推进本迭代原型设计。

但是，以下功能存在较高技术风险，不能在本迭代承诺真实实现：

- arXiv API 论文抓取。
- PDF / HTML 内容解析。
- 结构化摘要自动生成。
- 带出处的智能问答。
- 知识图谱自动构建。
- 多 Agent 流水线和任务队列。

因此，本迭代应将这些功能处理为“静态展示 + Mock 数据 + 状态提示”，重点保证页面流程完整、用例覆盖清楚、技术风险记录明确。真实接口、解析算法、LLM 问答和 Agent 后台服务应放到后续技术原型迭代中验证。
